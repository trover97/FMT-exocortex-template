# Стратегия

> **Назначение:** твои **направления развития (R1, R2, …)** с бюджетами времени. Living document — обновляется при стратегической сессии (раз в неделю/месяц).

## Принципы

1. **R = направление, не задача.** «Освоить системное мышление», не «прочитать книгу».
2. **Бюджет в часах в неделю** — реалистичный, не желаемый.
3. **5-7 направлений** — больше не удержать в голове.
4. **Источник:** Lifework (миссия) → Dissatisfactions (что не так) → R{N} (куда направляюсь)

## Программный горизонт

> Ритм планирования среднесрочных инициатив (6-12 мес). Параметризуемый — пилот выбирает методологию на L3.

```yaml
 cycle_duration: "_не задано_"          # заполнить вручную: 6w, 12w, 1q
 cycle_type: "time-boxed"              # альтернативы: rolling, milestone-based
```

**Текущий цикл:** _название / фокус_
**Следующий вех:** _что должно быть готово к концу цикла_

### 6-week cycles (Shape Up)

> Применять вместо недельного time-boxing, если:
> - РП >40h aggregate бюджет
> - Требует deep work (архитектура, исследование, дизайн)
> - Не зависит от еженедельных сигналов

Формат: 6 недель = 1 cycle. IWE-адаптация Shape Up (basecamp.com/shapeup):
- **shape** (до цикла) — формулировка проблемы и подхода
- **bet/pitch** — решение, что берём в цикл
- **build** (6 недель) — реализация
- **cool-down** — интеграция, документация, планирование следующего

CyclePlan вместо WeekPlan на период цикла.

### Rolling OKR

> Адаптивный ритм пересмотра OKR — не по календарю, а по триггерам.

**Триггеры пересмотра:**
- 2+ недели подряд miss по ключевой метрике
- Новый P0-сигнал меняет фокус
- Результат эксперимента инвалидирует текущую гипотезу

**Формат пересмотра:**
1. Проверить: триггер устойчивый (не шум)?
2. Если да — создать draft `inbox/okr-review-YYYY-MM-DD.md` с предложением изменений (что меняем, на что, обоснование)
3. Уведомить пилота (Telegram): триггер + draft + запрос подтверждения
4. Только пилот (или агент по явному подтверждению пилота) — обновить `docs/Strategy.md` §«Квартальный фокус»
