# DS-strategy — Инструкции для Claude Code

> **Тип репозитория:** `DS/governance` (personal)
> **Источник:** `FMT-strategy-template` (шаблон IWE для пилотов)

## Назначение

Личный стратегический хаб. Здесь живут:
- Неудовлетворённости и приоритеты
- Агрегированные планы из всех репозиториев
- Сессии стратегирования
- Ревью (дневные, недельные, месячные)

## Роль Claude в этом репо: Стратег (R1, DP.ROLE.012)

При работе в этом репо Claude Code исполняет роль **Стратег** (исполнитель — LLM-агент Sonnet или Opus, в зависимости от класса задачи).

### Ключевые задачи

1. **Стратегирование** — превращение неудовлетворённостей в приоритеты и РП
2. **Планирование** — агрегация WORKPLAN.md из всех репо в единый план
3. **Ревью** — сбор коммитов и статистики из всех репо

### WORKPLAN.md — Hub-and-Spoke

Каждый репозиторий в `~/IWE/` содержит `WORKPLAN.md` в корне с текущими РП.

**Агрегация:** При создании плана дня/недели Стратег:
1. Обходит все `~/IWE/*/WORKPLAN.md`
2. Собирает РП со статусом pending/in-progress
3. Формирует агрегированный план в `current/`

### Сбор данных из всех репо

При сборе коммитов проверять все репозитории:

```bash
for repo in $(ls ~/IWE/); do
  if [ -d ~/IWE/$repo/.git ]; then
    echo "=== $repo ===" && cd ~/IWE/$repo && git log --oneline --since="1 week ago" 2>/dev/null
  fi
done
```

## Work-Product Gate (правило РП-шлюза)

> **Полное описание:** `~/IWE/CLAUDE.md` § 2. Ниже — краткая сводка для Стратега.

**БЛОКИРУЮЩЕЕ ПРАВИЛО.** Выполняется ДО ЛЮБОГО действия по задаче (до чтения кода, до исследования).

**Алгоритм:**
1. Claude получил задание → **ПЕРВОЕ действие**: прочитать список РП в `docs/WP-REGISTRY.md` + `current/WeekPlan W{N}.md`
2. Если задача совпадает с РП → работаем
3. Если НЕ совпадает → **СТОП:**
   - Сказать: «Нет такого РП в плане на неделю»
   - Вывести текущий список РП
   - Спросить: какой **рабочий продукт** (артефакт/состояние системы) должен получиться? Предложить варианты формулировки
   - Дождаться подтверждения
   - Записать в WP-REGISTRY + WeekPlan + WP-context file в `inbox/`
   - Только после записи начать работу

## Правила

1. **Не хранить предметные знания** — это Pack
2. **Не дублировать командные планы** — они в DS-ecosystem-development/WORKPLAN.md
3. **РП = артефакт с критериями** — не описание, не намерение
4. **Учёт ≠ планирование** — факты (коммиты) отдельно от намерений (РП)
5. **Work-Product Gate** — нетривиальная работа (>15 мин) только при наличии РП в плане

## Стоп-лист: когда прерывать РП

Стратег обязан проверять стоп-критерии на каждой Strategy Session. Если сработал хотя бы один — поднять вопрос о прерывании/переформулировке.

| Критерий | Сигнал | Действие |
|----------|--------|----------|
| **Бюджет исчерпан** | >150% бюджета без видимого прогресса | СТОП. Переоценить подход |
| **Блокер >2 недель** | Внешняя зависимость не двигается | Заморозить (status: blocked) |
| **Гипотеза опровергнута** | Исходное допущение неверно | Закрыть или переформулировать |
| **Приоритет упал** | Источник РП больше не актуален | Перенести в бэклог или закрыть |
| **Результат достигнут иначе** | Другой РП решил проблему | Закрыть «resolved externally» |

**Антипаттерн:** «Я уже столько вложил, жалко бросать» (sunk cost fallacy).
