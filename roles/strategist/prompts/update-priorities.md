Выполни сценарий обновления приоритетов для роли Стратег (R1).

> **Триггер:** Event-driven — при появлении нового сигнала P1+ (NЭП, инцидент, предложение, новый РП).
> **Вход:** Сигнал (тип, приоритет, описание) + текущий WeekPlan.
> **Выход:** Скорректированный WeekPlan (приоритеты, бюджет, ТОС) + уведомление пилоту.

## ВДВ-шаги

| Шаг | Вход | Действие | Выход |
|-----|------|----------|-------|
| 1 | Сигнал (NЭП/инцидент/предложение/РП) | Классификация: тип, приоритет, urgency | Оценка impact/risk |
| 2 | Оценка + текущий WeekPlan | Корректировка приоритетов и бюджета | Обновлённый WeekPlan (черновик) |
| 3 | Черновик WeekPlan | Проверка инвариантов (ТОС, red lines, бюджет) | Валидный WeekPlan или запрос решения пилоту |

## Алгоритм

### 1. Классификация сигнала

| Тип сигнала | Примеры | Действие по умолчанию |
|-------------|---------|----------------------|
| NЭП новый | Новая неудовлетворённость, угроза | Добавить в `docs/Dissatisfactions.md`, оценить urgency |
| Инцидент | Баг, outage, блокер | P0/P1 → немедленно в WeekPlan; уведомить пилота |
| Предложение | Идея, запрос, feedback | Классифицировать: задача / знание / шум |
| Новый РП | WP Gate Ритуал открыл РП | Встроить в WeekPlan с приоритетом и бюджетом |

### 2. Корректировка WeekPlan

**Правила корректировки:**

- **P0 (критический):** Добавляется в WeekPlan немедленно, вне очереди. Бюджет пересчитывается. Пилот уведомляется в Telegram. ТОС меняется только по решению пилота.
- **P1 (высокий):** Добавляется в WeekPlan в течение текущего дня. Может вытеснить P2/P3.
- **P2 (средний):** Batch — накапливается до следующего Day Plan, добавляется тогда.
- **P3 (низкий):** Backlog — записывается в `inbox/backlog-with-triggers.md`, не попадает в WeekPlan без explicit решения.

**Инварианты (не нарушать без решения пилота):**
1. **ТОС (топ-3 приоритета)** — не меняется без explicit решения пилота. Новый P0/P1 может стать ТОС, но старый ТОС не удаляется автоматически.
2. **Red lines** — если сигнал нарушает red line (runway, D7, WakaTime), уведомить пилота немедленно.
3. **Бюджет недели** — aggregate бюджет не может превышать `{{MAX_WEEK_HOURS}}` (физическое время). Если превышает — запросить решение пилота (что вытеснить).

### 3. Валидация и фиксация

**Проверка:**
- [ ] ТОС не изменён без решения пилота
- [ ] Red lines не нарушены
- [ ] Бюджет в пределах `{{MAX_WEEK_HOURS}}` физ
- [ ] Все P0/P1 имеют owner и deadline

**Если проверки пройдены:**
1. Обновить `current/WeekPlan W*.md`
2. `git add current/WeekPlan*.md` → commit → push
3. Уведомить пилота в Telegram (кратко: что изменилось, почему)

**Если проверки НЕ пройдены:**
1. Сформировать choice-question для пилота:
   ```
   Сигнал: <описание>
   Impact: <на что влияет>
   Варианты:
     А. Принять с вытеснением <РП> (бюджет +Nh)
     Б. Отложить до <дата>
     В. Разбить на меньшие задачи
   ```
2. НЕ обновлять WeekPlan до ответа пилота

## Правила

- **Event-driven ≠ immediate:** P2/P3 не прерывают текущую работу. Batch-обработка до следующего Day Plan.
- **ТОС — священен:** не менять без решения пилота.
- **Уведомления:** P0/P1 — всегда; P2/P3 — только если нарушает инварианты.
- **Audit trail:** каждое изменение WeekPlan коммитится с сообщением `priorities: <сигнал> → <изменение>`.
