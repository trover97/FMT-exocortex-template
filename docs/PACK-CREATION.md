# Создание Pack: от нуля до рабочей базы знаний

> Pack — репозиторий с формализованными знаниями предметной области.
> Это единственный source-of-truth для доменных знаний в IWE.
> Source-of-truth процесса создания: [SPF](https://github.com/TserenTserenov/SPF)

---

## Зачем нужен Pack

Без Pack Claude каждый раз «не знает» специфику твоей области — термины, методы, failure modes. С Pack это знание загружается в контекст автоматически.

Генеративная цепочка (подробно: [principles-vs-skills.md](principles-vs-skills.md)):

```
FPF (мета-онтология)  →  SPF (фреймворк Pack)  →  Pack (домен)  →  DS (реализация)
```

Pack живёт между SPF и DS: SPF задаёт структуру, Pack заполняет её доменным знанием.

---

## Шаг 0: отличить домен от темы

Pack создаётся для **домена**, не для темы.

| Домен | Тема (не для Pack) |
|-------|-------------------|
| Разработка ML-систем | Машинное обучение |
| Системный анализ | Системное мышление |
| Управление продуктом | Продуктовое мышление |
| Технический дизайн API | REST API |

**Тест:** есть ли люди, которые ДЕЛАЮТ это профессионально? Что они производят (артефакты)?
Если да — домен. Если «интересуюсь» — тема.

## Быстрый старт: `/pack-new`

Написать в Claude Code:

```
/pack-new
```

или просто: **«хочу создать пак»**, **«новый пак по теме X»**, **«создай Pack для Y»**.

Скилл проведёт через 5 шагов:

| Шаг | Что происходит |
|-----|---------------|
| 0 | Проверяет наличие FPF и SPF — клонирует при необходимости |
| 1 | 3 вопроса: кто практикует? что производят? как ошибаются? |
| 2 | Предлагает 2-3 варианта имени → ты выбираешь |
| 3 | Заполняет bounded context (что входит/не входит/термины) |
| 4 | Создаёт структуру `PACK-{slug}/` со стартовыми файлами |
| 5 | Показывает дорожную карту наполнения Ф1-Ф6 |

## Структура Pack

```
PACK-{slug}/
├── README.md                      # Название + одна строка о домене
├── REPO-TYPE.md                   # Тип: Pack, upstream: FPF + SPF
├── CLAUDE.md                      # Инструкции для агента
├── 00-pack-manifest.md            # Метаданные + entity index
├── ontology.md                    # Термины домена (Ubiquitous Language)
├── 01-domain-contract/
│   ├── 01A-bounded-context.md     # Что входит/не входит/термины
│   └── 01B-distinctions.md        # Ключевые различения (A.7 FPF)
├── 02-domain-entities/            # Роли, методы, WP — перечень
├── 03-methods/                    # Карточки методов практики
├── 04-work-products/              # Карточки рабочих продуктов
├── 05-failure-modes/              # Типичные ошибки и сигналы
├── 06-sota/                       # Состояние области (опционально)
└── 07-map/                        # Карта связей между сущностями
```

Шаблон: [SPF/pack-template](https://github.com/TserenTserenov/SPF/tree/main/pack-template)

## Именование Pack

**Формула:** `PACK-{slug}` где slug — существительное-домен на латинице, kebab-case.

**Критерии имени (все обязательны):**
1. Специфично: исключает соседние домены (`product-management`, а не `management`)
2. Широко: включает ядро методов, не один инструмент (не `jira`)
3. Узнаваемо: практик сразу понимает, о чём это

**Примеры:** `PACK-product-management`, `PACK-system-analysis`, `PACK-technical-writing`

**Антипримеры:** `PACK-everything`, `PACK-jira`, `PACK-notes`, `PACK-ideas`

## Что делать после создания: дорожная карта Ф1-Ф6

Pack не заполняется за один присест. Наполнение — итеративное.

**Порядок фаз по ценности:**

| Фаза | Файл | Суть | Время | Приоритет |
|------|------|------|-------|-----------|
| **Ф1. Различения** | `01B-distinctions.md` | 7-10 пар «X ≠ Y» домена (FPF A.7) | 1-2ч | Первая — без неё остальное ненадёжно |
| **Ф2. Сущности** | `02-domain-entities/` | Роли, WP, методы — только перечень | 1-2ч | Вторая — даёт карту |
| **Ф3. Методы** | `03-methods/` | Входы, выходы, критерии качества | 2-4ч | Третья — ядро Pack |
| **Ф4. Рабочие продукты** | `04-work-products/` | Артефакты + Definition of Done | 1-2ч | Четвёртая |
| **Ф5. Failure modes** | `05-failure-modes/` | 5-10 FM: причина + сигнал | 1ч | Высокая ценность, часто пропускают |
| **Ф6. SoTA** | `06-sota/` | Источники, версия знания | 1-2ч | Последняя, опционально |

**Принцип:** начни с Ф1 (различения). Первые 5 различений уже дают ценность — Claude перестаёт путать термины домена.

## Как наполнять Pack в процессе работы

Не нужно заполнять Pack «с нуля» в один сеанс. Правильный подход — захватывать знания по ходу работы:

```
Обнаружил паттерн или различение → /ke → Capture: [что] → PACK-{slug}/01B-distinctions.md
```

**Команда `/ke` (Knowledge Extraction):**
- Вызвать на рубеже сессии или когда найдено важное различение
- Claude анонсирует: «Capture: [что] → [куда]»
- При накоплении 5+ captures — запустить `/apply-captures`

## Связь Pack с DS

Pack содержит **«что это»** (знания домена).
DS содержит **«как мы это делаем»** (реализации, код, планы).

```
Pack-{slug}/03-methods/           ← метод как таковой (домен)
DS-{проект}/docs/how-we-do-X.md  ← как этот метод реализован у нас
```

Нельзя хранить в Pack: код, конфиги, планы, реестры. Это идёт в DS.
Нельзя хранить в DS: определения, различения, методы, failure modes. Это идёт в Pack.
