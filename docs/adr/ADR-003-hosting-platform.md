# ADR-003: Платформа-хостинг IWE

**Статус:** Принято (спецификация)
**Дата:** 2026-03-29
**Контекст:** WP-180 Ф4, FMT-exocortex-template, DP.EXOCORTEX.001

---

## Контекст

IWE сейчас доступна только через Git+CLI: `gh repo fork` → `bash setup.sh`. Это барьер для пользователей без опыта работы с терминалом (кейс Egor'а: «не знаю Git, не знаю CLI»). При этом ценность IWE — в методологии и агенте, а не в форме доставки.

**Из обратной связи участников:**
- Участник #2: нужен вход без Git, через веб
- Egor: хочет попробовать «одной кнопкой», потом разобраться
- abcdef0101: VM-образ как третий способ доставки

**Аналогия:** WordPress.org (сам разворачиваешь) vs WordPress.com (хостинг). Оба дают один продукт, разные точки входа. Пользователь WordPress.com может экспортировать всё и перейти на self-hosted.

## Проблема

1. **Барьер входа.** Git + CLI + fork + setup.sh ≈ 30–60 мин для новичка с риском ошибок.
2. **Нет управляемого онбординга.** setup.sh задаёт 7 вопросов сразу — нет постепенного входа.
3. **Нет обновления без потерь** (решено в Ф0.5–Ф1, но только для CLI-пользователей).

## Решение

### Два слоя доставки (не «два продукта»)

```
Слой A: Дистрибутив (текущий)        Слой Б: Платформа-хостинг (новый)
──────────────────────────           ──────────────────────────────────
gh repo fork + bash setup.sh         Веб → аккаунт → агент-онбордер
Git = хранилище и история             Git = хранилище (скрыт)
Пользователь управляет файлами        Агент управляет файлами от имени пользователя
Обновление: bash update.sh            Обновление: агент предлагает автоматически
```

Оба слоя работают с одним и тем же дистрибутивом (FMT-exocortex-template). Платформа-хостинг — это **тонкий слой** поверх, не отдельный продукт.

### Архитектура платформы-хостинга

```
┌─────────────────────────────────────────────────────────┐
│  Веб-интерфейс (чат)                                    │
│  • Агент-онбордер: настраивает IWE под пользователя     │
│  • Агент-партнёр: обычная работа через чат              │
│  • /iwe-update: агент предлагает обновления             │
├─────────────────────────────────────────────────────────┤
│  API хостинга                                           │
│  POST /setup?level=T1   — bash setup.sh --level=T1 --yes│
│  POST /update           — bash update.sh --yes          │
│  GET  /export           — git bundle (полный экспорт)   │
├─────────────────────────────────────────────────────────┤
│  Git-репозиторий пользователя (приватный)               │
│  • Тот же FMT-exocortex-template fork                   │
│  • Хранится на платформе (или GitHub пользователя)      │
└─────────────────────────────────────────────────────────┘
```

### Градиентный вход — уровни установки

Пользователь переходит по уровням в своём темпе (не по неделям):

| Уровень | Что входит | Что делает setup.sh | Время входа |
|---------|------------|---------------------|-------------|
| **T1** | Claude Code + CLAUDE.md + memory/ | Шаги 1-3 (placeholders, CLAUDE.md, memory) | ≤15 мин |
| **T2** | + ритуалы ОРЗ + extensions/ + params.yaml | + шаг 4b (skills, hooks) + шаг 6 (DS-strategy) | +30 мин |
| **T3** | + Pack + бот | + инструкции по Pack + бот-онбординг | +1h |
| **T4** | + роли + launchd | + шаг 5 (роли, автоматизация) | +1h |

**Ключевой принцип:** каждый следующий уровень — не замена предыдущего, а добавление. `setup.sh --level=T2` дополняет T1-установку.

### Скриптуемый API (для хостинга)

Хостинг управляет IWE через те же скрипты, но без интерактивных промптов:

```bash
# Первичная установка уровня T1 (без вопросов)
bash setup.sh \
  --level=T1 \
  --yes \
  GITHUB_USER=alice \
  EXOCORTEX_REPO=my-exocortex \
  WORKSPACE_DIR=/home/alice/IWE

# Обновление
bash update.sh --yes

# Повышение уровня
bash setup.sh --level=T2 --yes
```

`--yes` — флаг для хостинга: пропустить все интерактивные промпты, использовать defaults.

### Экспорт данных

Пользователь платформы-хостинга не теряет данные при уходе:

```bash
# Полный экспорт: git clone = все файлы + история
git clone https://hosting.iwe.ai/alice/my-exocortex ~/IWE/my-exocortex
cd ~/IWE/my-exocortex && bash setup.sh --level=T4
```

Это гарантирует vendor lock-in = 0.

### Сценарий С6: VM-образ (из обратной связи abcdef0101)

```bash
vagrant up  # скачивает образ Ubuntu + tmux + Claude Code + IWE T1
# → пользователь в tmux-сессии в ~/IWE/
```

Vagrant-образ = третий способ доставки (для DevOps-аудитории, тестирования).

## Что нужно реализовать (backlog)

|---|----------|---------|-------------|
| 1 | `setup.sh --yes` — неинтерактивный режим | Платформа | WP-180 Ф4 ✅ (`--level` добавлен) |
| 2 | `setup.sh --level=TX` — повышение уровня | Платформа | WP-180 Ф4 ✅ |
| 3 | Веб-интерфейс (чат → агент-онбордер) | Хостинг | WP-189 (MCP) |
| 4 | API хостинга (POST /setup, POST /update, GET /export) | Хостинг | Отдельный WP |
| 5 | Vagrant-образ | Доставка | Отдельный WP |

## Решение не принято: «v2 с нуля»

**Отклонено (27 мар 2026).** IWE = дистрибутив (rolling release, не мажорные версии). WP-180 реализует SOTA инкрементально. Двойной бюджет нереалистичен при 60h/нед. Выбрана инкрементальная стабилизация.
