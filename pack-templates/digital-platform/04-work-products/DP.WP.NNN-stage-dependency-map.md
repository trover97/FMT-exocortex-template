# Stage Dependency Map

## 1. Определение

**Stage Dependency Map** — формат рабочего продукта Аналитика ограничений, описывающий план работы по устранению идентифицированного ограничения системы. Это **dependency graph без временного измерения**:

- **Узлы** = этапы. Каждый этап содержит одну или несколько параллельных работ / РП.
- **Рёбра (внутренние)** = жёсткая зависимость. «Следующий этап начинается только после завершения предыдущего».
- **External-рёбра** = зависимости от работ вне target-системы. Указывают на конкретный РП / репо / внешнего поставщика.

## 2. Назначение

Без формата ответ Аналитика «делать сначала X, потом Y» оставляет вопросы: «X и Y могут идти параллельно? X блокирует только начало Y или весь Y? Что делать с зависимостью от другого РП?». Stage Dependency Map даёт явный ответ.

Артефакт служит:
- Пилоту — карта работы на ближайшие N этапов с явными границами
- Артефактору — вход для декомпозиции каждого этапа на физические артефакты с приёмкой
- Стратегу — input для НЭП-обсуждения

## 3. Различия с похожими форматами

| Формат | Что есть | Чего нет в Stage Dependency Map |
|--------|----------|----------------------------------|
| **PERT** | Узлы=события, рёбра=задачи с оценками времени | Оценки времени, critical path |
| **Gantt** | Задачи по календарной шкале | Календарь, абсолютные даты |
| **Kanban** | Колонки status (todo/doing/done) с WIP-limits | Status-колонки, WIP-limits |
| **CCPM Network** | Dependency + buffer management | Feeding buffers, project buffer |

Stage Dependency Map — **чистая структура зависимостей**.

## 4. Структура

### YAML-схема

```yaml
stage_dependency_map:
  target: WP-NNN | project-name | rcs-profile:<account_id>
  constraint_ref: "<ссылка на Constraint Brief>"
  generated_at: YYYY-MM-DD
  generated_by: <твой DP.ROLE.NNN> / .qwen/skills/bottleneck-pick

  stages:
    - id: stage_1
      label: "<краткое имя этапа, существительное-артефакт>"
      goal: "<какое состояние достигается этим этапом>"
      parallel_works:
        - "<работа A>"
        - "<работа B>"
        - "<работа C — может быть РП-ссылкой WP-NNN>"
      external_deps:
        - target: "WP-MMM в другом репо"
          required_state: "<что должно быть достигнуто там>"

  edges:
    - from: stage_1
      to: stage_2
      type: hard_dependency
      reason: "<почему stage_2 не может начаться без stage_1>"
```

```markdown

- работа A

**External-зависимости:** нет

### → Этап 2: <label>
> Может начаться только после завершения Этапа 1 (причина: <reason>)

**Параллельные работы:** ...

**External-зависимости:**
- WP-MMM в другом репо (требуется: <required_state>)
```

## 5. Правила построения

### Обязательные

1. **Каждый этап имеет цель.** Не «сделать X», а «достигнуто состояние Y».
2. **Параллельные работы внутри узла действительно параллельны.** Если работа B зависит от работы A — они в разных этапах.
3. **Edge = жёсткая зависимость.** Не «желательно сначала», именно «иначе не сможем».
4. **External-зависимости указывают конкретный target.** Не «зависит от другого РП», а «зависит от WP-MMM, требуется status=...».
5. **Без дат, часов, дней.** Формат структурный, не временной.
6. **Без status-колонок (todo/doing/done).** Это задача Kanban.
7. **Этап = существительное-артефакт.** «Активация N пилотов», не «Активировать пилотов».

### Запрещённые

- Циклы (Stage A → Stage B → Stage A) — топологическая сортировка обязательна
- Edges с типом, отличным от `hard_dependency`
- Date-fields, resource-fields, status-fields

## 6. Пример

**Target:** WP-NNN (зонтичный РП первой когорты), bottleneck = operational gap (activation).

```markdown
## Stage Dependency Map: WP-NNN

### Этап 1: Активация N ранних пилотов
**Цель:** ≥50% пилотов совершили первое целевое действие (открыли репо + получили руководство + ответили на 1 вопрос).

**Параллельные работы:**
- Доставка персонального руководства (через managed-канал)
- Отправка вечернего сообщения с инструкцией активации
- Check-in от Навигатора (опциональный, не блокирует)

**External-зависимости:**
- WP-MMM (Аттестатор daily-run) — должен работать стабильно

### → Этап 2: Wave-rollout
> Может начаться только после завершения Этапа 1 (причина: без подтверждённой conversion ≥50% на ранних пилотах следующая волна даст ту же operational-gap-картину).

**Параллельные работы:**
- Bulk-onboarding SQL для wave-2
- Подготовка broadcast-сообщений wave-2
- Solo-smoke pre-flight перед каждой волной

**External-зависимости:**
- WP-KKK Ф17.9 TG-broadcast — должен быть оттестирован

### → Этап 3: Когорта-mass, daily-rhythm стабилизация
> Может начаться только после завершения Этапа 2 (причина: rhythm работает только когда есть mass для daily-ритма).

**Параллельные работы:**
- Daily ping-bot
- Weekly check-in от Навигатора
- Monthly retro

**External-зависимости:** нет
```

## 7. Использование Артефактором (handoff)

После получения Stage Dependency Map от Аналитика, Артефактор берёт каждый этап и применяет к нему свой метод декомпозиции:

| Что делает Аналитик | Что делает Артефактор |
|----------------------|------------------------|
| Stage 1: «...» — параллельные работы | Stage 1 → стадии с физическими артефактами на входе/выходе каждой |
| Параллельные работы как список | Каждая работа → чеклист приёмки |
| External-зависимость как target | Артефактор детализирует, какое именно состояние нужно |

Sequencing: Аналитик (куда смотреть) → Артефактор (как сделать).
